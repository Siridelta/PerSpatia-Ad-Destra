# PerSpatia 基础架构 (Infrastructure)

本文记录项目的基础设施决策。

---

## 1. 路由方案 (Routing)

### 1.1 HashRouter
*   **决策**：全面采用 `HashRouter` 而非 `BrowserRouter`。
*   **理由**：
    *   **环境适配**：适配 `live-server`、GitHub Pages 等静态托管环境，确保在没有后端重写规则的情况下，移动端刷新页面不丢失路径。
    *   **快速迭代**：在开发变体（Variants）时，Hash 路由能更简单地通过 URL 共享特定场景。

## 2. CSS 架构与样式工程惯例 (CSS Architecture Practice)

针对现代前端框架环境下的样式组织，我们试验使用以下“混合架构”实践方案，以解决全局样式污染、字体字重散乱等工程痛点：

### 2.1 动态与极高频属性：内联样式 (Inline Styles)
*   **适用场景**：需要每帧渲染或随手势高频变动的位置信息（如相机拖拽、节点随动坐标 `transform: translate(x, y)`）。
*   **实践**：直接通过 `style={{...}}` 挂载在 DOM 上。坚决禁止用 CSS 类名去控制此类高频数据，以最大化绕过浏览器的样式重算开销。

### 2.2 静态与复杂结构：CSS Modules (局部作用域)
*   **适用场景**：常规组件的排版、颜色、静态动画、发光阴影等复杂 CSS 声明。
*   **实践**：逐步将散落的全局 `.css` 文件重构为 `xxx.module.css`。这样既能保留手写极其复杂的、与数学或发光相关的自定义样式的自由度，又能通过编译期生成的哈希类名彻底杜绝组件间的相互污染。

### 2.3 设计令牌与全局控制：CSS Variables (CSS 变量)
*   **适用场景**：全应用统一的字体、色彩氛围场、间距等。
*   **实践**：在顶层入口定义类似 `--spatia-font-family`, `--spatia-font-weight`, `--theme-ambient-color` 的全局变量。
*   **联动优势**：设置面板或相机距离计算出的参数，只负责动态修改这些根节点的 CSS 变量，底层的所有 CSS Modules 组件只需消费（引用）这些变量，从而实现牵一发而动全身的统一控制。
